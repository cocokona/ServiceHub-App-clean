## 1. 离线缓存策略

### 1.1 三层缓存架构

| 层级 | 技术方案 | 访问延迟 | 适用场景 | 容量 |
|------|----------|----------|----------|------|
| L1 Memory | React Context / Zustand 内存状态 | < 1ms | 当前页面数据、会话状态 | 10MB |
| L2 LocalStorage | SQLite (via Expo SQLite) + AsyncStorage | ~10ms | 离线持久化数据、用户偏好 | 200MB |
| L3 Remote | Supabase PostgreSQL | ~200ms | 权威数据源、多端同步 | 无限制 |

**读取顺序**: L1 → L2 → L3（cache-aside 模式）
**写入顺序**: L3 → L2 → L1（write-through 模式）

### 1.2 缓存过期策略

| 数据类型 | TTL | 过期行为 | 刷新触发 |
|----------|-----|----------|----------|
| 服务目录 (services) | 24h | 后台静默刷新 | 应用启动 + 每日一次 |
| 订单列表 (jobs) | 5min | 即时失效 | 状态变更 / 用户主动刷新 |
| 用户 Profile | 30min | 登录时刷新 | 登录 / 更新后 |
| 技师列表 | 15min | 切换类别刷新 | 类别切换 / 搜索 |
| 聊天记录 (messages) | 实时 | 不缓存 | Realtime 订阅 |

**实现**: 每个缓存条目记录 `cachedAt` 和 `ttl`。读取时检查 `Date.now() - cachedAt > ttl`，过期则标记为 stale，后台触发 refresh。

### 1.3 存储容量管理

- **总配额**: 200MB (LocalStorage 上限)
- **预警阈值**: 150MB → 触发 LRU 淘汰
- **关键淘汰**: 85% 占用时自动清理过期图片缓存
- **紧急处理**: 95% 占用时暂停自动同步，只保留核心数据（用户、订单、消息）
- **已满处理**: 提示用户清理缓存，禁用离线功能

**LRU 实现**: 使用 `accessOrder` Map 记录最近访问时间，淘汰最旧条目。

### 1.4 降级处理（三级）

| 级别 | 场景 | 处理方式 |
|------|------|----------|
| L1 | 缓存命中，网络不可用 | 显示缓存数据 + 顶部灰色离线提示横幅 |
| L2 | 缓存未命中，网络不可用 | 显示骨架屏 + 刷新按钮 + 重试提示 |
| L3 | 缓存损坏 / 数据异常 | 空状态页 + 手动重试引导 + 客服入口 |

## 2. 数据同步机制

### 2.1 在线/离线状态切换

```
Online  →  实时读写 Supabase + 后台同步 LocalDB
Offline →  写入 LocalDB 队列，标记待同步
恢复    →  批量队列同步 → 增量拉取 → 冲突解决 → 更新 L1/L2
```

**状态检测**: `NetInfo` 监听网络变化，200ms 防抖避免状态抖动。
**状态广播**: 通过 AppContext 全局共享，所有页面订阅。

### 2.2 增量同步

```typescript
// 基于 last_sync_at 时间戳的增量拉取
async function syncJobs(userId: string, lastSyncAt: string) {
  const { data } = await supabase
    .from('jobs')
    .select('*')
    .eq('customer_id', userId)
    .gt('updated_at', lastSyncAt)
    .order('updated_at', { ascending: true });
  return data;
}
```

| 数据类型 | 增量策略 |
|----------|----------|
| jobs | `updated_at > last_sync_at` 获取变更 |
| messages | 按 `job_id` + `created_at` 增量拉取 |
| profiles | 登录时全量 + 变更时 Supabase Realtime 推送 |
| services | 目录变化少，全量刷新即可 |

### 2.3 冲突检测与解决

| 冲突类型 | 策略 | 说明 |
|----------|------|------|
| 同字段更新 | Last Write Wins | 时间戳优先，后写入覆盖 |
| Checklist 状态 | 逐条合并 | 逐条对比，勾选状态取最新 |
| 离线队列操作 | 先入先出 | 顺序同步，确保操作一致性 |
| 乐观锁冲突 | version CAS | 更新时校验 version，失败则重新读取 |

### 2.4 重试机制

| 重试次数 | 延迟 | 超时 | 行为 |
|----------|------|------|------|
| 第1次 | 立即 | 2s | 直接重试 |
| 第2次 | 2s | 3s | 指数退避 |
| 第3次 | 5s | 5s | 最终失败，上报日志 |
| 后续 | 放弃 | — | 提示用户手动重试 |

**队列管理**: 失败操作进入 `pending_sync` 队列，网络恢复时批量执行。

### 2.5 同步状态实时反馈

| 状态 | 视觉反馈 | 行为 |
|------|----------|------|
| 同步中 | 顶部蓝色横幅 + 旋转动画 | 不可进行冲突操作 |
| 同步完成 | 绿色 Toast (2秒自动消失) | 提示用户最新数据已更新 |
| 同步失败 | 红色弹窗 + 重试按钮 | 提供手动重试入口 |
| 离线模式 | 灰色横幅 + 离线图标 | 提示当前为离线状态 |

## 3. 页面一致性规范

### 3.1 离线状态数据展示

- **列表页**: 显示缓存数据，无操作限制，下拉可刷新
- **详情页**: 显示缓存数据，编辑项标记为草稿状态，保存时加入同步队列
- **顶部横幅**: 灰色背景 `#F0F0F0` + 离线图标 + "当前离线，显示缓存数据" 提示

### 3.2 加载状态规范

| 场景 | 加载样式 | 颜色 |
|------|----------|------|
| 下拉刷新 | 自定义旋转指示器 | 蓝色 `#014DB2` |
| 页面首次加载 | 骨架屏 (3-5个 shimmer 脉冲条) | 灰色 `#EBEEF5` |
| 按钮提交 | 替换为 spinner + 禁用按钮 | 蓝色 `#014DB2` |
| 分页加载 | 底部水平线 + 圆点指示器 | 蓝色 `#014DB2` |

### 3.3 错误提示规范

| 错误类型 | 提示方式 | 颜色 |
|----------|----------|------|
| 网络错误 | 红色 Toast + 重试按钮 | 红色 `#EF4444` |
| 认证失败 | 自动跳转登录页 + 清空缓存 | 红色 `#EF4444` |
| 数据异常 | 空状态页 + 客服入口 | 琥珀色 `#F59E0B` |
| 服务器超时 | 自动降级读缓存 + 提示稍后重试 | 琥珀色 `#F59E0B` |

### 3.4 同步指示器规范

| 位置 | 样式 | 状态映射 |
|------|------|----------|
| 顶部 | 蓝色横幅 + 旋转动画 + 同步提示文字 | 同步中 |
| 列表底部 | 水平线 + 圆点指示器 | 加载更多中 |
| 卡片右下角 | 小圆点 (8px) | 蓝色=同步中, 绿色=已同步, 灰色=未同步 |
| 页面全局 | 离线横幅 | 灰色=离线状态 |

## 4. 边界场景处理

### 4.1 弱网环境

| 场景 | 检测 | 处理 |
|------|------|------|
| 2G/3G | `NetInfo` 类型检测 | 禁用图片预加载，纯文字优先展示 |
| DNS 超时 | 请求超时 > 5s | 切换 HTTP DNS + 本地 hosts 缓存 |
| 429 限流 | HTTP 429 响应 | 指数退避 5s-60s，最大等待 1 分钟 |
| 连接频繁断开 | 30s 内断开 3 次 | 标记为弱网，降低同步频率 |

### 4.2 存储空间不足

| 占用率 | 行为 |
|--------|------|
| 85% | 触发图片/缓存自动清理，保留最近 7 天 |
| 95% | 暂停自动同步，只保留核心数据（用户、订单、消息） |
| 100% | 提示用户清理缓存，禁用离线功能，仅在线模式 |

### 4.3 多设备并发同步

| 策略 | 实现 |
|------|------|
| 乐观锁 | `version` 字段，更新时 CAS 校验，失败则重读重试 |
| 单会话 | 强制登录，踢出其他设备会话，JWT 唯一绑定 |
| 实时推送 | Supabase Realtime 订阅多端变更，即时推送 |
| 最终一致性 | 离线队列顺序执行，确保操作顺序一致 |

### 4.4 其他异常场景

| 异常 | 检测 | 处理 |
|------|------|------|
| 服务端 504 超时 | HTTP 504 | 自动降级读缓存，提示用户稍后重试 |
| 本地数据损坏 | 校验和失败 | 自动全量拉取修复，清空本地缓存 |
| 设备时间异常 | 与服务端时间差 > 5min | 以服务端时间为准，标记偏差日志 |
| 多用户切换 | 检测到新用户登录 | 清空旧用户缓存，重新拉取新用户数据 |

## 5. 可扩展性设计

### 5.1 新增页面接入

1. 在 `cacheConfig.ts` 中注册新页面的缓存策略（TTL、粒度）
2. 在 `syncConfig.ts` 中注册同步规则（增量字段、冲突策略）
3. 复用 `OfflineBanner`、`SyncIndicator`、`LoadingSkeleton` 等通用组件
4. 通过 `useOfflineData` hook 统一获取离线/在线数据

### 5.2 新增数据类型

```typescript
// cacheConfig.ts 示例
interface CacheConfig {
  key: string;
  ttl: number;           // 毫秒
  granularity: 'full' | 'incremental';
  eviction: 'lru' | 'fifo' | 'none';
  offlineFallback: 'cache' | 'placeholder' | 'empty';
}

const cacheRegistry: Record<string, CacheConfig> = {
  jobs: { ttl: 5 * 60 * 1000, granularity: 'incremental', eviction: 'lru', offlineFallback: 'cache' },
  services: { ttl: 24 * 60 * 60 * 1000, granularity: 'full', eviction: 'none', offlineFallback: 'cache' },
  // 新增数据类型只需在此注册
};
```
