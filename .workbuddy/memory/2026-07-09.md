## Checkout "Your profile was not found" 错误修复（根因：缺失 profiles 行 → FK 23503）
- **根因**：已登录用户缺少 `profiles` 行（账号早于 `on_auth_user_created` 触发器创建，或 profile 丢失）。`createOrderInProgress` 写入 `order_in_progress.customer_id` 触发 FK 23503，旧提示"sign out and sign in again"具误导性（signIn 并不创建 profile）。
- **修复（应用层自愈）**：
  - `database.service.ts` 新增 `ensureProfile()`：有会话则查 profiles，缺失则用 auth metadata 自插（受 `profiles_insert_self` RLS 允许），幂等。
  - `createOrderInProgress` 23503 分支：先 `ensureProfile()` 自愈后重试一次 INSERT，仍失败才抛原消息（保持对外错误契约不变）。
  - `Checkout.handlePay` 下单前调 `ensureProfile()`；`AppNavigator` 用户会话建立时主动 `ensureProfile()`（保护所有 FK 写入：jobs/messages/reviews）。
  - `auth.service.getCurrentUser`/`refreshSession` 加 `ensureProfile()` 自愈，避免缺失 profile 被误判为未登录。
- **修复（数据层）**：新增 `supabase/migrations/00006_backfill_profiles.sql`，为所有缺 profile 的 `auth.users` 补建 profiles 并幂等重建触发器，末尾 `NOTIFY pgrst`。
- 测试：补 `supabase.auth.getSession` 与 builder `maybeSingle` mock；`npm test` 44 全过，`tsc --noEmit` 零错误。
- **约定提醒**：服务层重构不得改变既有导出函数签名/返回/抛出 message（本任务严格保留原 23503 提示文案）。

## 技师分配（Select Technician）功能修复
- **需求**：客户下单页从数据库拉取真实技师、显示在"选择技师"区、仅可选 active 技师、选中正确持久化并在保存订单时校验+落库。
- **根因/缺陷**：
  1. 数据绑定丢失：`ServiceDetails.handleNext` 只存 `technicianName/Avatar` 到 bookingData，从不存 `technicianId`；`Job` 类型与 `mapDbJobToAppJob`/`mapDbOrderToAppJob` 均无 `technicianId`；`createOrderInProgress`/`createJob` 的 INSERT 不写 `technician_id` 列 → 选中技师从不以 FK 落库。
  2. 缺"active"过滤：`fetchTechnicians` 仅 `role=technician AND deleted_at IS NULL`，无激活态过滤，停用技师仍可选。
  3. 无校验+脏预选：`preferredTech` 若为本地 `recommendedTechnicians`（非库内真实 id）且 helpers 加载后未匹配，脏选择残留。
- **修复**：
  - 新增 `supabase/migrations/00007_add_profiles_is_active.sql`：`profiles` 加 `is_active BOOLEAN NOT NULL DEFAULT true` + 部分索引 + `NOTIFY pgrst`。
  - `fetchTechnicians` 加 `.eq('is_active', true)`（DB 级过滤）。
  - `types/index.ts` `Job` 增 `technicianId?`；两个 mapper 映射 `technician_id`；`createOrderInProgress`（含 23503 重试 INSERT）与 `createJob` 写入 `technician_id`。
  - `ServiceDetails`：loading/empty 状态；`handleNext` 校验必须选中有效技师并写入 `technicianId`；helpers 加载后清除不在 active 列表内的脏 `preferredTech`。
  - `Checkout.handlePay`：保存前校验 `bookingData.technicianId` 存在，缺失则拦截告警；传入 `technicianId`。
- **验证**：`npm test` 46 全过（新增 2 用例：is_active 过滤、technician_id 持久化+回映射），`tsc --noEmit` 零错误。
- **注意**：迁移 00007 必须在目标 Supabase 执行后方可生效（否则 `is_active` 列查询报错）。

## 导航修复：NAVIGATE 'Jobs' 未被任何 navigator 处理
- **报错**：`The action 'NAVIGATE' with payload {"name":"Jobs"} was not handled by any navigator`（开发期警告，但导航实际失败，按钮无响应）。
- **根因**：`ServiceCompletion` 是 Stack 屏（与 `TechnicianTabs` 平级），而 `Jobs` 是**嵌套在 `TechnicianTabs` 内部的 Tab 屏**（Stack → TechnicianTabs → Jobs，属于孙级）。`navigation.navigate('Jobs')` 在 Stack 中找不到该屏，跨兄弟 navigator 分支无法解析。
- **修复**：`src/screens/technician/ServiceCompletion.tsx` 两处 `navigation.navigate('Jobs')` → `navigation.navigate('TechnicianTabs', { screen: 'Jobs' })`，显式回到 Tab navigator 并激活 Jobs 页（同时自动 pop 回栈底）。
- **审计**：grep 全 `src` 的 `navigate(` 调用，其余目标（`ServiceDetails`/`Checkout`/`Tracking`/`SupportChat`/`JobDetails`/`ActiveService`/`ServiceCompletion`）均为 Stack 直接屏，可正常冒泡解析，无其它损坏引用。
- **验证**：`npx tsc --noEmit` 零错误。
